# The Multi-Hop Investigator – Adaptive RAG System

## Overview
The Multi-Hop Investigator is an Agentic RAG system designed to answer multi-step queries requiring dependent retrieval and verification. Unlike linear RAG pipelines, this system utilizes a bounded iterative loop to decompose objectives, verify intermediate findings, and "pivot" search strategies based on discovered entities.

The system is built on **LangGraph** to ensure stateful, controlled execution and to mitigate the "RAG Triad" failure modes (Retrieval, Augmentation, and Generation).

---

## Core Engineering Features
* **Stateful Orchestration & Research Ledger:**
    Managed via a structured `AgentState` schema to ensure data integrity across the lifecycle:
    * `sub_queries`: The execution roadmap generated by the Planner.
    * `current_documents`: Temporary storage for retrieved context awaiting grading.
    * `research_ledger`: A repository of facts updated only when `relevance_score >= threshold`, ensuring all stored data is grader-validated.
    * `retry_count`: Incremented by the Grader to enforce loop exit conditions.
* **Query Decomposition (Planner Node):**
    Analyzes high-level prompts to generate a sequence of atomic sub-tasks for focused retrieval.
* **Adaptive RAG with Self-Correction:**
    * **LLM-Based Retrieval Grader:** Quantifies factual sufficiency against a 0.8 relevance threshold.
    * **Rewriter Node (`rewriter_node`):** Reformulates queries based on Grader feedback to resolve information gaps or temporal inconsistencies.
* **Multi-Hop Pivot Logic:**
    Identifies bridge entities in initial search results to dynamically initiate dependent sub-queries.
* **Execution Guardrails:**
    * **Circuit Breaker:** Logic-gate that aborts execution upon detection of invalid or contradictory premises.
    * **Loop Control:** Hard-coded `MAX_REVISIONS = 3`. If retries are exhausted, the system returns a bounded response indicating insufficient verified data rather than generating speculative output.

---

## Technical Architecture & State Mutation
The system implements a cyclic **StateGraph** with explicit node contracts:

1. **Planner Node (`planner_node`)**: Mutates state by populating the `sub_queries` list.
2. **Retriever Node (`retriever_node`)**: Fetches raw context and updates `current_documents`.
3. **Grader Node (`grader_node`)**: Evaluates docs; writes `relevance_score` and increments `retry_count`.
    * *Conditional Edge*: If `score < 0.8` & `retry_count < 3` → **rewriter_node**.
    * *Conditional Edge*: If `score >= 0.8` → **generator_node**.
4. **Rewriter Node (`rewriter_node`)**: Mutates the current query string for the next retrieval attempt.
5. **Generator Node (`generator_node`)**: Synthesizes final output, restricted strictly to `research_ledger` entries to prevent the introduction of unverified information.
